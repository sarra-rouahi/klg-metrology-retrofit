# KLG Metrology — Web (Python + React)

Pont entre ton code Python métier (`jarreb.py`, OpenCV) et ton interface React :
**caméra externe réelle → backend Python → interface web moderne**, le tout sur ta machine.

## Architecture

```
klg-metrology/
├── backend/                  ← Python (FastAPI)
│   ├── metrology_core.py     ← logique métier (modèle K(z), calibration, mesures)
│   ├── camera.py             ← capture webcam réelle (OpenCV, thread, reconnexion auto)
│   ├── server.py             ← serveur API + stream vidéo + sert le frontend
│   └── requirements.txt
├── frontend/                 ← React (Vite)
│   ├── src/
│   │   ├── KLGMetrology.jsx  ← ton interface, branchée sur l'API réelle
│   │   └── api.js            ← client API (fetch vers /api/*)
│   └── package.json
├── start.bat                  ← lancement Windows (double-clic)
├── start.sh                   ← lancement Linux/macOS
└── README.md
```

**Comment ça communique :**
- Le frontend affiche le flux de ta caméra externe en streaming (MJPEG) — c'est la vraie image, pas une simulation.
- Quand tu cliques sur la vidéo, calibres, ou changes de zoom, React appelle l'API (`/api/...`) du serveur Python.
- Le serveur Python fait tourner **exactement** la même logique que ton script original (modèle de calibration `numpy.polyfit`, linéaire/quadratique, interpolation K(z), persistance JSON/CSV) et répond avec le nouvel état.
- Tout est persisté sur disque dans `backend/data/` (`calibrations.json`, `measurements.json`, exports CSV) — identique au format de ton script d'origine.

## Installation (une seule fois)

Il te faut **Python 3.9+** et **Node.js 18+** installés sur le PC.

```bash
cd backend
pip install -r requirements.txt

cd ../frontend
npm install
npm run build
```

## Lancer l'application

**Windows** : double-clique sur `start.bat`
**Linux/macOS** : `./start.sh`

Ou manuellement :
```bash
cd backend
python server.py --camera 0
```
Puis ouvre ton navigateur sur **http://localhost:8000**

> `--camera 0` est l'index de ta webcam USB. Si tu as plusieurs caméras branchées
> (webcam interne du PC + caméra de métrologie externe), essaie `--camera 1`, `--camera 2`, etc.
> jusqu'à voir la bonne image. Tu peux aussi appeler `GET /api/cameras` pour lister
> les index détectés.

## Développement (modifier l'interface)

Si tu veux modifier `KLGMetrology.jsx` et voir les changements en direct sans
rebuilder à chaque fois :

```bash
# Terminal 1 — backend
cd backend
python server.py

# Terminal 2 — frontend en mode dev (hot reload)
cd frontend
npm run dev
```
Puis ouvre **http://localhost:5173** (le frontend en dev redirige automatiquement
les appels `/api/*` vers le backend sur le port 8000, voir `vite.config.js`).

Quand tu as fini de modifier, rebuild pour la version "production" servie par défaut :
```bash
cd frontend
npm run build
```

## Ce qui a été repris fidèlement de ton code Python original

- **`SmartCalibrationModel`** : même logique exacte — 2 points calibrés → modèle linéaire
  `K(z) = a·z + b`, 4+ points → modèle quadratique `K(z) = a·z² + b·z + c`, calculé avec
  `numpy.polyfit`, avec R² et résidus.
- **`CalibrationManager`** : sauvegarde/chargement JSON, archivage des anciennes
  calibrations, interpolation linéaire en secours si pas de modèle.
- **Mesures point-à-point** : clic P1 → clic P2 → distance en px/mm/µm, historique
  persisté, export CSV (même format `;` séparateur).
- **Tangentes (haut/bas/gauche/droite)** : même séquence de 4 clics → centre + largeur/hauteur.
- **Caméra** : capture via `cv2.VideoCapture`, thread dédié, reconnexion automatique,
  flip horizontal/vertical, calcul FPS — comme dans `CameraManager` d'origine.

## Différences avec le script original

| Original (jarreb.py)                  | Version web                                  |
|----------------------------------------|-----------------------------------------------|
| Fenêtre OpenCV (`cv2.imshow`)           | Flux MJPEG dans le navigateur                |
| Raccourcis clavier captés par OpenCV    | Mêmes raccourcis, captés par le navigateur (C, Z, G, T, R, H, I, Espace, Échap) |
| Mono-poste, un seul process             | Serveur web — accessible aussi depuis un autre appareil sur le même réseau (`http://<IP-du-PC>:8000`) |
| Affichage HUD dessiné sur les frames    | HUD dessiné en React (overlay au-dessus du flux vidéo) |

## Dépannage

- **"Backend inaccessible"** dans l'interface → le serveur Python n'est pas lancé, ou
  tourne sur un autre port. Vérifie le terminal où tu as lancé `server.py`.
- **"Caméra non détectée"** → vérifie le branchement USB, essaie un autre `--camera N`,
  ou regarde `logs/server.log` pour le détail de l'erreur OpenCV.
- **Le flux vidéo est noir/lent** → certaines webcams USB ont besoin d'un pilote DirectShow
  spécifique sous Windows ; le code essaie déjà `cv2.CAP_DSHOW` puis se rabat sur le
  backend par défaut.
- **Port 8000 déjà utilisé** → `python server.py --port 8001` puis ouvre `http://localhost:8001`.
